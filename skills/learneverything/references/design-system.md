# Дизайн-система vault

Красота = жёсткий шаблон на каждый тип документа + дисциплина типографики. Автор следует буквально, ревьюер проверяет формально. Скелеты — в `templates/`.

## Режимы оформления (из Профиля, вопрос «где будешь читать»)

- **Obsidian** — всё богатство: кастомные callouts (ниже), wikilinks `[[Понятие]]`, mermaid, граф связей.
- **Оба** — Obsidian-primary: те же wikilinks `[[Понятие]]` и кастомные callouts, что в режиме Obsidian. Портируемость вторична — граф связей важнее, чем идеальный рендер на GitHub. Wikilinks на GitHub деградируют в текст, это приемлемо.
- **Чистый md** — портируемое подмножество: пять стандартных GitHub alerts (`> [!NOTE]`, `[!TIP]`, `[!IMPORTANT]`, `[!WARNING]`, `[!CAUTION]`), обычные markdown-ссылки, сноски, mermaid.

**Граф связей — не украшение, а суть базы знаний. Файлы обязаны ссылаться друг на друга `[[wikilink]]` с самого начала (режимы Obsidian и оба), иначе Obsidian показывает набор несвязанных точек.** Минимальный каркас связей:
- `MOC.md` — центр (Map of Content): ссылается на `[[Профиль ученика]]`, `[[Карта темы]]`, `[[План обучения]]`, каждый источник и каждый урок. Создаётся сразу на Этапе 2, растёт по ходу.
- `План обучения` ↔ `[[Карта темы]]` ↔ файлы `[[01 Научная база]]` и т.д.; каждый модуль плана ссылается на свои уроки `[[Урок N — …]]`.
- Каждый урок в теле ставит `[[Понятие]]` на ключевые термины (Obsidian покажет их узлами сразу, даже до создания заметки — «призрачные» узлы резолвятся, когда notes-writer создаст сами заметки) и ссылки `[[Урок N−1 — …]]` / `[[План обучения]]`.
- Атомарные заметки в `Понятия/` — самый плотный слой связей (понятие ↔ понятие ↔ урок).

**Цвет графа (из `obsidian-preset/graph.json`).** Узлы окрашены по типу, чтобы граф читался структурно, а не серой кашей. Легенда:
- 🟢 бирюзовый — `Понятия/` (атомарные заметки, ядро знания)
- 🟠 оранжевый — `Уроки/`
- 🔵 синий — `Источники/`
- 🟣 фиолетовый — структура-хаб: `MOC`, `Карта темы`, `План обучения`
- ⚪ серый — служебное: `Профиль ученика`, `Терминология`, `Практические выводы`, `CLAUDE`

Группы завязаны на стандартные имена папок и файлов курса, поэтому работают в любом курсе без правок. Хочешь свою палитру — меняешь `rgb` (целое R·65536+G·256+B) в `graph.json`.
- **Notion** — локальные файлы пишутся как «чистый md», а при зеркалировании через Notion MCP адаптируются: callout-блоки Notion с иконкой и цветом вместо md-callouts (соответствие ниже), mermaid — код-блок с языком `mermaid` (Notion рендерит его сам), сноски `[^n]` превращаются в инлайн-ссылки `[название источника](url)` в месте цитирования, wikilinks — во внутренние ссылки на страницы Notion. Иерархия: страница сферы → подстраницы План, Источники, Уроки, Понятия, Терминология, Практические выводы; каждый урок и каждое понятие — отдельная подстраница.

Соответствие callouts → Notion: pretest — 🧭 фиолетовый, myth — 🔥 красный, recall — 🧠 синий, core — ⚓ зелёный, outside — 🌐 оранжевый, next — ➡️ серый. Правило «жирный = слой выжимки» действует и в Notion.

## Callouts (режим Obsidian)

Кастомные типы под механики обучения, стили в CSS-сниппете пресета:

| Синтаксис | Назначение |
|---|---|
| `> [!pretest] Проверь интуицию` | 2–3 вопроса до контента |
| `> [!myth] Миф и разбор` | refutation: миф → почему неверно → правильная модель |
| `> [!recall] Проверь себя` | retrieval-вопросы с местом для ответов |
| `> [!core] Что запомнить` | Cornell-итог: 3–5 ёмких строк, ядро урока |
| `> [!outside] Этого не было в твоих источниках` | ответ через веб-поиск сверх согласованного списка |
| `> [!next] Куда дальше` | 2–3 ранжированные рекомендации |

Не больше 2–3 типов callout на урок сверх обязательных по скелету. Callouts заменяют жирный и эмодзи как средство выделения.

## Типографика

- Один h1 на файл, дальше только h2–h3.
- Абзацы 2–4 предложения. Стена текста длиннее 5 предложений — разбить.
- Таблицы только для сравнений (2+ объекта по 2+ признакам), не для перечислений.
- **Жирный зарезервирован под слой Progressive Summarization:** жирное в конспекте = главное при повторном чтении. Скольжение по жирному должно читаться как выжимка урока. Ничего другого жирным не выделять.
- Сноски вместо голых URL: в тексте `[^1]`, блок сносок внизу файла. Голых URL в тексте быть не должно.

## Mermaid

- Одно направление на весь vault: `graph LR` (если сфере объективно нужно иное — задать в Профиле и держать везде).
- Максимум 8–10 узлов на схему; сложнее — две маленькие схемы.
- Единая палитра через `%%{init: {'theme':'neutral'}}%%`.
- Схема обязана нести содержание (структура, поток, причинность). Декоративные схемы запрещены.
- К каждой абстрактной концепции урока — одна содержательная схема рядом с текстом (dual coding).

## Картинки и видео

Схема (mermaid) — костяк и первый выбор всегда. Настоящая картинка и встроенное видео — только когда схема принципиально не передаёт нужное.

**Когда настоящая картинка (управляется полем «Картинки в уроках» в Профиле):**
- Только если поле = «нужны». Поле = «хватит схем» → картинок не ищем и не вставляем, обходимся mermaid и видео-ссылками.
- Только когда схема не тянет: реальный вид объекта (орган, тип кожи, порода животного, устройство, микрофото), реальные пропорции/текстура, кадр движения. Абстракцию, поток, причинность рисует схема — для них картинка не нужна.
- Каждая картинка заработана и относится к конкретному месту текста, не для украшения. Хорошее правило — не больше одной ключевой картинки на крупный раздел.

**Источник и права (любая тема):**
- Только свободно-лицензионное: Wikimedia Commons, Openverse, public domain, открытые атласы и учебники, CC-BY/CC0. Никаких случайных картинок из гугла с неясной лицензией.
- Автор проверяет каждую через WebFetch: файл открывается, показывает именно заявленное, лицензия свободная. Не подтвердилось — не вставляет.

**Синтаксис в уроке:**
```markdown
![Осмысленный alt-текст, что на картинке](https://upload.wikimedia.org/…/файл.jpg)
*Подпись: что показано. Источник: Wikimedia Commons, автор, лицензия CC-BY-SA 4.0.*
```
- `alt`-текст обязателен и осмысленный (доступность + подстраховка, если картинка не загрузится).
- Строка-подпись курсивом под картинкой: что показано + источник + лицензия.
- Автор ставит внешний URL (скачивать файлы он не умеет). Локализацию в `Уроки/приложения/` делает дирижёр отдельным шагом (см. SKILL, Этап 5) — грепает `![...](http…)`, качает, переписывает на локальный путь; видео (`[текст](url)` без `!`) не трогает.

**Видео в теле:**
- Ссылку с таймкодом (`…?t=1098` для YouTube) можно ставить прямо в нужном месте текста, когда движущаяся картинка нужна для понимания именно здесь: «посмотри движение диафрагмы 2:10–2:40 → [ссылка]». Это в дополнение к блоку «Куда дальше», а не вместо него.
- Видео тоже проверяется на живость и релевантность; помечать источник, как у сносок.

**Режимы:** локальные картинки в `Уроки/приложения/` и стандартный `![]()` работают во всех режимах. В Notion при зеркалировании картинки загружаются/встраиваются в блок изображения, видео — как embed.

## Шаблоны по типам

В `templates/` лежат скелеты с комментариями-инструкциями `<!-- ... -->`:

- `Урок.md` — полный скелет урока со всеми блоками в фиксированном порядке
- `Профиль ученика.md` — что и как записывать из интервью
- `Атомарная заметка.md` — одна идея = одна страница в `Понятия/`
- `MOC.md` — карта сферы: группы понятий со связями и пояснением, почему связаны
- `Терминология.md` — словарь сферы
- `Практические выводы.md` — как применять к жизни, накапливается по модулям

Комментарии-инструкции при генерации удаляются, в готовых файлах их быть не должно.
